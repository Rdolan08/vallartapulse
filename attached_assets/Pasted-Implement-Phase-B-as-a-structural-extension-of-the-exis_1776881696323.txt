Implement Phase B as a structural extension of the existing event system, not a rewrite. The goal is to resolve the double-count problem BEFORE any zone-aware pricing meaningfully changes rates.

OBJECTIVE
Build a deseasonalization step so event timing can be separated from observed comp pricing. This allows future zone multipliers to be applied safely. For this phase:
- preserve current behavior by default
- enable the plumbing for deseasonalized-baseline math
- optionally allow limited zone application behind a feature flag
- do NOT activate property-type math yet
- do NOT change any existing request shape
- do NOT remove or replace the current Phase A audit contract

KEY PRINCIPLE
Current comp output already reflects:
- same-neighborhood comp filtering
- actual stay-window quotes
- event-aware market pricing for those dates

Therefore:
DO NOT apply zone/property multipliers directly on top of current comp_output.

Instead:
1. compute current event stay multiplier from Phase A event audit
2. deseasonalize comp_output back to a neutral baseline
3. reapply event timing cleanly
4. optionally apply zone multiplier
5. leave property multiplier at 1.00 for now

FORMAL MODEL
Let:
- P_observed = current comp-engine recommendedRate
- M_event_avg = weighted average event multiplier across the stay window from Phase A audit
- P_base = deseasonalized baseline
- M_zone_avg = weighted average zone multiplier across the stay window (Phase B, optional)
- M_property_avg = 1.00 in this phase

Compute:
- P_base = P_observed / M_event_avg
- P_reconstructed = P_base * M_event_avg
- P_final = P_base * M_event_avg * M_zone_avg * M_property_avg

Invariants:
- if zone multipliers are disabled or 1.00, P_final must equal current P_observed within floating-point tolerance
- if no event applies, M_event_avg = 1.00 and P_base = P_observed

7) INSERTION POINT IN CURRENT PRICING PIPELINE
Find the existing stay-window pricing path in /api/rental/comps after:
- comp selection
- comp quote aggregation
- recommendedRate computation
- current seasonal/event overlay calculation

Current conceptual flow is something like:
1. build comp set
2. aggregate comp prices
3. compute recommendedRate
4. compute seasonal block / event overlay
5. return response

Change it to:
1. build comp set
2. aggregate comp prices
3. compute recommendedRate => this is P_observed
4. compute Phase A eventAudit using existing logic
5. compute deseasonalized baseline from recommendedRate + eventAudit
6. compute final overlay from eventAudit + neighborhood
7. compute finalRecommendedRate:
   - if ENABLE_EVENT_DESEASONALIZATION is false:
     finalRecommendedRate = recommendedRate
   - if ENABLE_EVENT_DESEASONALIZATION is true AND ENABLE_EVENT_ZONE_MULTIPLIERS is false:
     finalRecommendedRate = deseasonalizedBaseline * eventAvgMultiplier
     // must equal recommendedRate
   - if both flags true:
     finalRecommendedRate = deseasonalizedBaseline * finalOverlayMultiplier
8. return both the final rate and the enriched event audit block

This means:
- rollout 1 with zone OFF should be a no-op mathematically
- rollout 2 can enable limited zone effects safely
