Canonical shared schema

Use one shared module such as src/shared/events/eventRules.ts or a DB table with the same contract. Every pricing path should read from this and nowhere else.

export type EventType =
  | "holiday_peak"
  | "domestic_surge"
  | "circuit_identity"
  | "luxury_maritime"

  | "zona_romantica"

  | "conchas_chinas"
  | "centro"
  | "5_de_diciembre"
  | "marina_vallarta"
  | "nuevo_vallarta";

  | "condo"
  | "villa"
  | "luxury";

  property_type: PropertyTypeKey;
  multiplier: number;        // property-type effect on top of event + zone
  notes?: string;
};
export type EventRule = {
  event_key: string;
  display_name: string;
  year: number;
  core_start_date: string;   // YYYY-MM-DD
  core_end_date: string;     // YYYY-MM-DD
  shoulder_start_date?: string;
  shoulder_end_date?: string;
  core_multiplier: number;
  shoulder_multiplier?: number;
  event_type: EventType;
  application_mode: ApplicationMode; // default: per_night_weighted
  event_impact_zones: EventImpactZone[];
  property_type_adjustments?: PropertyTypeAdjustment[];
  // auditability
  active: boolean;
  priority: number;          // higher wins if same-night conflict
  date_confidence: "confirmed" | "derived" | "tentative";
  source_refs: string[];
  notes?: string;
};

Why this shape works

It keeps the system:

* spatially aware
* asset aware
* auditable
* easy to test

It also cleanly separates three things:

* event intensity via core_multiplier / shoulder_multiplier
* where it matters via event_impact_zones
* what inventory responds differently via property_type_adjustments

Zone-multiplier philosophy

This is the intended meaning of the zones in the seed set:

* zona_romantica: strongest for Pride and BeefDip because that cluster is the center of LGBTQ nightlife and south-side visitor concentration. The area is widely described as a tourist-heavy, LGBTQ-friendly district, and “Old Town / South Side” language is common in travel references.  ￼
* amapas: immediate hillside spillover behind Los Muertos, so it benefits heavily from south-side identity and nightlife demand.
* conchas_chinas: more luxury spillover than street-level nightlife effect.
* centro and 5_de_diciembre: secondary beneficiaries for core-city events.
* marina_vallarta and nuevo_vallarta: strongest fit for the Mexico Open because Marina Vallarta is tied to golf and marina activity, while the 2026 tournament is explicitly at VidantaWorld Vallarta / Nuevo Nayarit.  ￼

Keep the comp engine intact. This is an overlay.

const compResult = runCompEngine(input);

const finalRecommendedRate =

Event overlay contract

  date: string;
  event_key?: string;
  phase?: "core" | "shoulder";
  base_event_multiplier: number;
  zone_multiplier: number;
  property_type_multiplier: number;
  final_night_multiplier: number;
};
export type EventOverlayResult = {
  applied: boolean;
  stayMultiplier: number;
  nights: EventOverlayNight[];
  winningEvents: string[];
};

Night-by-night computation rules

For each night in the stay:

1. Find matching active event rules.
2. Determine whether the night falls in core or shoulder.
3. Start with core_multiplier or shoulder_multiplier.
4. Apply the matching neighborhood zone multiplier.
5. Apply the property-type multiplier.
6. If multiple events overlap on the same night, use the highest-priority event, then highest resulting multiplier as the tiebreaker.
7. Average nightly multipliers across the stay.

Formula:

nightMultiplier =
  baseEventMultiplier
  * zoneMultiplier
  * propertyTypeMultiplier;
stayMultiplier =
  sum(nightMultiplier for each night) / totalNights;

That gives you exactly what you asked for: per_night_weighted, spatially aware, asset aware, and auditable.

Modules that should migrate to this shared logic

These should stop owning their own event timing:

* current date-based event multiplier helper
* any seasonality/date overlay module that hardcodes event windows
* stay pricing calculator
* pricing explanation / audit response builder
* any tests or fixtures with embedded event date logic

In plain English: if any module says “if date between X and Y, bump rate,” that logic needs to move into the shared event rules source.
