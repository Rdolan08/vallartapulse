Your read is right.

The proposal was directionally strong, but it overreached in one important way: it treated the event layer like a fresh pricing engine instead of an additive extension to what you already shipped. And yes, the double-count problem is the blocker.

My revised position

Do not implement zone and property multipliers yet.

First, formalize the existing event system into an auditable shared schema without changing the math. Then resolve, on paper, whether future spatial logic applies to:

* the current comp-engine output, or
* a deseasonalized baseline derived from comps

Until that is answered, any location-aware event uplift is dangerous.

The actual structural issue

You called it exactly:

If comps already use:

* same-neighborhood filtering
* actual stay-window quotes
* event-aware market pricing for those dates

then a second event overlay on top of that can absolutely double-count, and in some cases triple-count.

So the key design decision is:

Option A

Overlay applies to current comp output

* simplest
* lowest engineering effort
* high double-count risk
* probably only safe for weak shoulder shaping or explanatory audit output

Option B

Overlay applies to deseasonalized comp baseline

* architecturally correct
* requires new work
* only path that makes strong zone/property multipliers defensible

I agree with you that the long-term answer is B.

What I would do now

Phase A: additive schema upgrade only

No pricing behavior change.

Extend the existing PV_EVENT_OVERLAYS / EventOverlay structure with optional fields:

export const EVENT_RULES_SCHEMA_VERSION = 1;
export interface EventOverlay {
  key: string;
  name: string;
  startDate: string;
  endDate: string;
  shoulderStartDate?: string;
  shoulderEndDate?: string;
  pct: number;
  shoulderPct?: number;
  priority?: number;
  dateConfidence?: "confirmed" | "derived" | "tentative";
  sourceRefs?: string[];
  notes?: string;
  // reserved for later phases
  eventImpactZones?: Array<{
    neighborhoodKey: string;
    multiplier: number;
    notes?: string;
  }>;
  propertyTypeAdjustments?: Array<{
    propertyType: "condo" | "villa" | "luxury";
    multiplier: number;
    notes?: string;
  }>;
}

Important part:

* eventImpactZones and propertyTypeAdjustments can exist in schema now
* but remain unused until the double-count question is resolved

That gives you forward compatibility without pretending the math is settled.
